## 五、发现的问题与待优化项

### 5.1 【严重】多页面未过滤已禁用部门员工

**现状**：员工档案和考勤管理已实现过滤已禁用部门员工，但以下模块**未做过滤**：

| 模块 | 后端 API | 问题描述 |
|------|---------|---------|
| 绩效管理 | `GET /api/performance/` | 查询所有员工，未排除已禁用部门 |
| 薪资核算 | `POST /api/salary/calculate/{period}` | 核算所有员工，包含已禁用部门 |
| 薪资核算 | `GET /api/salary/check-completeness/{period}` | 完整性检查包含已禁用部门 |
| 薪资核算 | `GET /api/salary/results/{period}` | 结果展示未过滤 |
| 社保公积金 | `GET /api/social-insurance/` | 查询所有员工，未排除已禁用部门 |
| 报表导出 | `GET /api/reports/roster` | 花名册导出包含已禁用部门 |
| 报表导出 | `GET /api/reports/attendance/{period}` | 考勤表导出未过滤 |
| 报表导出 | `GET /api/reports/salary/{period}` | 薪资表导出未过滤 |
| 报表统计 | `GET /api/reports/stats` | 员工总数统计未排除 |

**修复方案**：

在每个涉及 `db.query(Employee)` 的地方，添加已禁用部门过滤逻辑：

```python
# 统一过滤模式
disabled_dept_ids = db.query(SysDictBase.id).filter(
    SysDictBase.category == 'department',
    SysDictBase.is_enabled == False
).all()
if disabled_dept_ids:
    query = query.filter(Employee.department_id.notin_([d[0] for d in disabled_dept_ids]))
```

**建议**：提取为公共工具函数，避免重复代码：

```python
# app/core/query_utils.py
def filter_active_employees(query, employee_model=Employee):
    """过滤掉已禁用部门的员工"""
    from app.models.models import SysDictBase
    disabled_ids = db.query(SysDictBase.id).filter(
        SysDictBase.category == 'department',
        SysDictBase.is_enabled == False
    ).all()
    if disabled_ids:
        return query.filter(employee_model.department_id.notin_([d[0] for d in disabled_ids]))
    return query
```

**涉及修改的文件**：
- `backend/app/api/performance.py` — `get_performances()`、`export_performances()`
- `backend/app/api/salary.py` — `check_data_completeness()`、`calculate_salary()`、`get_calculation_results()`
- `backend/app/api/social_insurance.py` — `get_social_insurance()`、`export_social_insurance()`
- `backend/app/api/reports.py` — `get_stats()`、`export_roster()`、`export_salary()`、`export_attendance()`

### 5.2 【中等】数据模型单文件过大

**现状**：所有 20+ 个模型定义在 `backend/app/models/models.py`（约 440 行）。

**建议**：按业务域拆分：
- `models/system.py` — SysUser, SysRole, SysPermission, SysDictBase, SysLog
- `models/employee.py` — Employee, EmployeeSalary, EmployeeSalaryAdjustment
- `models/attendance.py` — AttendanceRecord
- `models/salary.py` — SalaryCalculation, CalculationLog, SalarySlip, PerformanceScore, SocialInsurance 等
- `models/approval.py` — ApprovalRecord

**优先级**：低，当前可工作，不影响功能。

---

### 5.3 【中等】前端 API 层缺失

**现状**：前端页面直接在组件中使用 `axios` 调用后端 API，没有统一的 API 封装层。

**问题**：
- 相同 API 路径在多处硬编码
- 请求/响应处理逻辑重复
- 难以统一修改 base URL 或添加拦截器

**建议**：按项目规则创建 `frontend/src/api/` 目录，按模块封装 API 调用。

---

### 5.4 【低】Vite 代理端口与 Docker 端口不一致

**现状**：
- `vite.config.js` 代理目标：`http://127.0.0.1:8001`
- `.env` 配置后端端口：`BACKEND_PORT=8010`
- Docker 内后端端口：`8010`

**影响**：本地开发时 `npm run dev` 无法代理到后端（端口不匹配）。

**修复**：将 `vite.config.js` 中代理目标改为 `http://127.0.0.1:8010`。

---

### 5.5 【低】后端端口配置不一致

**现状**：
- `backend/Dockerfile` 暴露端口 `8010`
- `docker-compose.yml` 映射 `${BACKEND_PORT}:8010`
- `.env` 中 `BACKEND_PORT=8010`
- 但 `vite.config.js` 代理目标是 `8001`

**修复**：统一为 `8010`。

---

### 5.6 【低】员工导出接口未过滤已禁用部门

**现状**：`GET /api/employees/export` 没有像列表接口那样默认排除已禁用部门员工。

**修复**：在 `export_employees()` 中添加与 `get_employees()` 相同的禁用部门过滤逻辑。

---

### 5.7 【建议】薪资核算公式文档化

**现状**：薪资核算逻辑在 `salary.py` 的 `calculate_salary()` 中，约 150 行计算代码，缺少公式说明。

**核算流程**：
1. 取最新薪资档案（按 effective_date）
2. 取月度考勤（出勤率）
3. 取月度绩效（系数）
4. 取月度社保公积金
5. 如有月中调薪，使用折算后工资
6. 计算应发 = (折算基本工资 + 补贴 + 提成) × 出勤率 + 实际绩效 × 系数 × 出勤率
7. 扣除社保公积金个人部分
8. 计算实发

## 八、关键业务逻辑

### 8.1 部门禁用机制

- 数据字典 `sys_dict_base` 中 `category='department'` 的记录有 `is_enabled` 字段
- 禁用部门后，该部门员工在以下场景被排除：
  - 员工档案列表 ✅ 已实现
  - 考勤管理列表 ✅ 已实现
  - 绩效管理 ❌ 未实现
  - 薪资核算 ❌ 未实现
  - 社保公积金 ❌ 未实现
  - 报表导出 ❌ 未实现

### 8.2 薪资核算流程

```
数据完整性检查 → 执行核算 → 计算实发(个税) → 提交审批 → 导出
     ↓              ↓            ↓              ↓
  检查各数据源   汇总计算应发   累计预扣法计税   主管→经理两级
```

### 8.3 钉钉同步

- 花名册同步：全量/增量模式，按钉钉 userId 匹配
- 考勤同步：按月同步，映射钉钉考勤列到系统字段
- 部门同步：从钉钉获取一级部门同步到数据字典

### 8.4 审批流程

- 两级审批：主管审核 → 经理审核
- 提交后薪资记录状态变为"审核中"
- 驳回后状态变为"已驳回"
- 每级可独立通过/驳回
